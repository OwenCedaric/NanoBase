Issue 01 // CLI Architecture Est. 2025

CLI的重构：
面向人类与
AI代理的双重架构。

By: @ghchinoy, @Saboo_Shubham_ & Zack Akil

核心哲学
The Core Philosophy

绝大多数在今年构建的CLI，最终都会被自动化Agent调用。但它们并没有准备好。

交互式提示、彩色输出和终端UI（TUI）已经成为CLI设计的标准假设，但当自动化Agent尝试解析结果时，这些全都崩溃了。然而，剥离这些功能会让原本人类的体验变得糟糕。

为了解决这个问题，你不必在两个受众之间做出取舍。你可以为两者同时设计CLI。

核心哲学非常简单：将数据与表示层解耦 (Decouple the data from the presentation)。当Agent或脚本调用你的CLI时，它需要原始的、结构化的数据（如JSON）。当人类调用它时，他们需要将这些数据渲染成可读的、交互式的格式（如TUI）。将你的CLI内部逻辑视为一个发射数据的“引擎”，而终端UI仅仅是其中一种可能的“客户端”，这样你就能无缝地服务于两者。

相同的 watch 命令应该为人类提供实时更新的TUI，同时为Agent提供结构化的NDJSON事件流： 

# Human gets the interactive TUI
a2acli watch --task abc123

# Agent gets structured NDJSON
a2acli watch --task abc123 --no-tui

你可以在不维护两个独立代码库的前提下实现这种无缝的双受众体验。采用以下这些经过深思熟虑的设计模式，让你的CLI对机器可预测，同时对人类保持愉悦：

01

结构化可发现性
Structured Discoverability

CLI的入口应该清晰地映射其能力。Agent和人类都是从 --help 开始的。

  • 按功能分组命令，而不是按字母顺序。像 任务管理、信息、配置 这样的分类，可以防止在根帮助输出中出现文字墙。
  • 明确标记入口点。在帮助描述中添加提示，如 (start here) 或 (typical first step)。Agent利用这些来决定首先调用哪个命令。

为每个命令填充三个字段：

短描述 (快速扫描)
单行摘要，5-10个词，以动词开头。
长描述 (深度理解)
命令的作用、为何使用、与相似命令的区别。
示例 (即时效用)
3-5个具体的可复制粘贴的示例。示例比描述更重要。开发者会先复制粘贴再阅读；Agent会解析示例以推断Flag模式。
Task Management:
  send        Send a message to start or continue a task (start here)
  watch       Subscribe to live updates for a running task
  get         Retrieve the current state of a task

Discovery:
  describe    Inspect an agent's identity, skills, and capabilities

外部化的Agent指令

可发现性超越了 --help。不要假设大语言模型 (LLM) 天生知道如何使用你的工具——请明确地教导它。
在仓库根目录发布一个 AGENTS.md 文件，为AI开发者定义交战规则、默认工作流和架构标准。对于复杂的命令，包含一个 skills/ 目录，其中包含外部Agent可以直接摄取的专用Prompt文件。

02

Agent优先的互操作性
Agent-first Interoperability

为了能被Agent使用，CLI必须是可解析且可预测的。

一切皆支持 --json

每一个产生数据的命令都应该支持 --json 或 --no-tui 标志。输出必须是有效的JSON或NDJSON。如果Agent无法解析你的输出，你的CLI在Agent的世界里就不存在。

自动侦测受众

支持 NO_COLOR 和 [APP]_NO_TUI 环境变量。当设置了这些变量，或者当标准输出 (stdout) 被管道重定向时，完全跳过交互元素。没有提示，没有加载动画，没有颜色代码。

非交互式降级

使用TUI（如 Bubble Tea）的命令应该具备一种纯净模式，向stdout发出标准文本或JSON。TUI是人类接口，JSON输出是Agent接口。同一个命令，两个渲染器。

保护上下文窗口

Agent具有严格的Token限制。主动截断巨大的文本块，并在默认输出中脱敏敏感机密，以免淹没Agent的上下文窗口或将API密钥泄露到对话日志中。如果Agent确实需要原始、未经过滤的有效载荷，要求其明确使用 --full 或 --verbose 标志。

数据预处理与排序

Agent不应该为了寻找重要信息而编写复杂的数据操作逻辑。自动预排序CLI输出，使得最关键、最可操作的项目（如：严重漏洞、无限制的密钥、待处理任务）始终出现在响应的最顶部。

委托状态管理

不要在CLI进程内将Agent困于交互式状态循环中。通过使用引用标识符（如 --task <ID>），保持CLI完全无状态。让后端服务维护长时间运行的Token上下文和对话历史，而CLI仅仅作为一种快速的传输机制。

03

配置与上下文
Configuration & Context

CLI应该了解其环境，而不需要在每次调用时都加上繁杂的标志。

遵循 XDG 标准

配置文件放在 ~/.config/app/config.yaml。主目录下不要放 dotfiles。不要使用带有专有格式的隐藏文件夹。

命名环境

支持通过简单的 --env 标志切换多个配置（local, staging, prod）。这对Agent尤其重要。编排器可以设置 --env prod 而无需知道具体的URL或Token。

# ~/.config/a2acli/config.yaml
default_env: "local"

envs:
  local:
    service_url: "http://127.0.0.1:9001"
  staging:
    service_url: "https://staging-agent.internal.corp"
    token: "my-staging-auth-token"
  prod:
    service_url: "https://agent.example.com"
    token: "my-secure-prod-token"
04

错误引导
Error Guidance

不要仅仅报告错误。提供一条解决路径。

上下文提示 (Contextual Hints)

当命令由于缺少先决条件而失败时，包含一行 Hint:：

Error: database not initialized
Hint: run 'a2acli init' to create the local database

Agent解析这些提示以进行自我纠正。人类则感激不必去翻阅文档。

快速失败 (Fail Fast)

在执行繁重逻辑之前验证配置和连接性。不要让Agent等待30秒的API调用后才发现认证Token丢失。

确定性的退出代码 (Deterministic Exit Codes)

  • 0: Success
  • 1: General error
  • 2: Invalid usage / bad flags
  • 3: Connection / auth failure

如果你的CLI在操作失败时返回0仅仅因为“命令本身运行了”，你会破坏每一个调用它的自动化系统。

05

标志与参数一致性

CLI的语法应当具有内在的可预测性。如果用户（或Agent）学会了如何使用一个命令，这种直觉应无缝延续到应用程序的其余部分。

  • 标准化简写：如果 -o 在一个命令中代表 --out-dir，它绝不能在另一个命令中代表 --output。
  • 位置 vs 可选：核心的必需实体使用位置参数（例如 a2acli get <task-id>），--flags 严格用于可选修饰符。
  • 安全的默认值：默认行为应始终是最安全、最常见的路径。破坏性操作必须强制要求明确的标志。
06

终端视觉设计

颜色应服务于功能目的，而非美学目的。

语义化色彩令牌

  • Accent: Headers, section labels
  • Command: Command names, flags
  • Pass: Success states
  • Warn: Transient, active tasks
  • Fail: Errors, rejected states
  • Muted: Metadata, types
  • ID: Unique identifiers

经验法则

  • 保留颜色用于状态指示。
  • 不要给描述或帮助文本上色。
  • 空白优于颜色来表达层次感（对齐方式比颜色更能传达结构）。
  • 同时支持亮色和暗色终端。
07

版本化与生命周期

CLI工具并非静态的伪像。它在动态序列中运行，在这些序列中版本会漂移，操作会突然中断，安全工作流需要审计跟踪。智能地处理这些生命周期事件可以防止自动化系统中的静默故障。

  • CLI应当知道自身的版本，并（可选地）通知用户自上次运行以来的重大更新。
  • 优雅地处理 SIGINT (Ctrl+C)。当人或Agent杀死一个正在运行的操作时，不能发生数据损坏。
  • 跟踪写入操作的"Actor"（源自git config或$USER），以确保可审计性。

Case Study: YouTube CLI

端到端的有效演示

以 YouTube CLI 为主要范例。你现在可以与Agent协同管理并审查你的YouTube频道。通过利用Agent感知的CLI技能，你和你的Agent可以从YouTube API构建出提取频道统计、切换视频可见性或上传新内容所需的确切工具集。

在以前，这需要寻找特定的MCP服务器或手动解析密集的YouTube API文档。

但要让AI Agent可靠地执行这样的工作流，底层的CLI必须为它而设计。如果 yt-cli 用一个交互式的“你确定要上传吗？(Y/n)”提示阻挡了Agent，或者返回了不可解析的彩色终端文本，整个自动化过程就会失败。

"2026年最好的CLI，
不是那些拥有最漂亮终端界面的。
而是那些在人类键入命令和Agent编程式调用时，表现得同样出色的CLI。"
