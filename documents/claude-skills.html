定义与分野

厘清核心概念：为何 Skills 不是另一种 MCP

01

MCP: 工具能力

通常，我们将 MCP (Model-Controller-Plugin) 定义为 Agent 的**工具能力**。它的核心是让模型能够访问外部系统、执行具体流程和获取关键信息。MCP 的边界可以很狭义，也可以很广义，这主要取决于具体实践中的定义。本质上，MCP 定位在**工具**——它更原子化，负责执行一个明确的任务。

可以将其理解为：命令行、自定义脚本、API 调用等一系列可执行单元的集合。

SKILLS: 技能抽象

技能（Skills）则是更高维度的概念，是 Agent 通过学习、训练或经验获得的综合**能力**。如果说过去的 Agent 依赖其内在的通用知识，那么 Skills 则提供了一个可供参考的、结构化的“强化学习路径”或“行为模板”。它描述的是如何运用多个工具和知识来完成一个更复杂的、带有策略性的目标。

本质区别
SKILLS
≠
MCP
MCP 定位在工具
Skills 定位在技能

正如 Anthropic 官方图示所表达的，Skills 并非要替换 MCP。工具是构成技能的基础，而技能则赋予了 Agent 更高的自主性和解决问题的能力。

架构的启发

在智能体和工具之间，增加一个新的抽象层

02

在过往的架构设计中，复杂的“技能”往往会演变成一个独立的子智能体（Agent as Tool / Agent as Skill），其技能详情则固化为子智能体的系统提示。这种设计增加了系统的复杂性。

若在智能体和工具之间
引入一个 技能层 (Skill Layer)，
我们将重新思考一个核心问题：

是否还需要额外引入一个子 Agent，
如果其核心能力可以被一个 Skill 清晰地承载？

新的架构思考
工具 (MCP) → 技能 (SKILLS) → 智能体 (AGENT)

实践与远方

从设计哲学到行动指南

03

友好的设计哲学

Skills 的设计对普通用户极为友好。要实现一个 MCP 工具，非编程科班出身的人可能会对诸多概念望而却步。而 Skill 的设计核心，无非是像写文档一样，清晰地讲明白一件事。为了执行脚本和代码，Anthropic 必然提供了一个虚拟执行环境，将技术复杂性转交给了容器，这本身就是一个值得关注的技术趋势。
